# 协议字段与联调检查清单

## BLE UUID

```text
Service:  6E400001-B5A3-F393-E0A9-E50E24DCCA9E
Write/RX: 6E400002-B5A3-F393-E0A9-E50E24DCCA9E
Notify/TX:6E400003-B5A3-F393-E0A9-E50E24DCCA9E
```

电脑端通过 Write/RX 下发命令，nRF52840 通过 Notify/TX 上传数据。

## 上行帧格式

```text
@<json_payload>*<crc16>\n
```

CRC 使用 CRC16-CCITT-FALSE：

```text
poly = 0x1021
init = 0xFFFF
输出 = 4 位大写十六进制
```

CRC 只计算 `<json_payload>` 的 UTF-8 字节，不包含 `@`、`*`、CRC 字符和换行。

校验向量：

```text
输入: 123456789
CRC: 29B1
```

## sample 示例

```json
{"msg":"sample","ver":1,"seq":12,"uptime_ms":53000,"temperature_c":36.5,"humidity_pct":58.2,"posture":"supine","battery_pct":87,"device_status":"monitoring","alarms":[]}
```

字段要求：

```text
msg              固定 sample
ver              固定 1
seq              设备侧递增序号
uptime_ms        设备上电后毫秒数，不要求 RTC
temperature_c    摄氏温度
humidity_pct     相对湿度百分比
posture          supine / side / prone / moving / invalid
battery_pct      0-100
device_status    normal / idle / monitoring / sensor_error / low_battery
alarms           没报警为 []，有报警可放 ["low_battery"]
```

## event 示例

```json
{"msg":"event","ver":1,"seq":13,"event_id":3,"uptime_ms":54000,"type":"kick","level":"warning","value":"-2.1C"}
```

字段要求：

```text
msg         固定 event
ver         固定 1
seq         每条 event 都用新的设备侧序号
event_id    同一次真实事件保持相同 id
uptime_ms   设备上电后毫秒数
type        kick / low_temp / prone_alarm / low_battery / sensor_error
level       info / warning / danger / error，第一版建议 warning
value       触发值，例如 -2.1C
```

关键事件连续重复 3 次：

```text
seq:      20, 21, 22
event_id: 5,  5,  5
type:     kick, kick, kick
```

电脑端会按 `event_id + type` 去重，只显示一次。

## 下行命令

```text
START\n
STOP\n
```

建议行为：

```text
START: monitoring = true，每秒上报 sample
STOP:  monitoring = false，停止上报但保持 BLE 连接
```

## 联调前自查

- [ ] 广播服务 UUID 是 NUS。
- [ ] 电脑端能扫描到设备。
- [ ] 连接后 NUS Notify 已启用。
- [ ] 收到 `START\n` 后才开始上报。
- [ ] 每秒至少发送 1 条 sample。
- [ ] 每条上行帧以 `@` 开头，以 `\n` 结尾。
- [ ] CRC 计算范围不包含 `@`、`*`、CRC 和换行。
- [ ] JSON 内字段名和枚举拼写完全一致。
- [ ] Notify 分片不会丢尾包。
- [ ] `STOP\n` 后停止上报，但 BLE 不主动断开。

## 常见问题

### 页面没有数据

检查：

1. 是否点击了“开始监测”。
2. nRF52840 是否收到 `START\n`。
3. NUS Notify 是否真正发送成功。
4. 帧末尾是否有 `\n`。
5. CRC 是否正确。

### 后端日志提示 CRC 失败

常见原因：

1. 计算 CRC 时包含了 `@`、`*`、CRC 或换行。
2. Notify 分片时丢了中间或尾部字节。
3. JSON 内容发送后又被修改。
4. 使用了非 UTF-8 字节。

### 事件重复显示

检查同一次事件的 3 次重复是否保持相同 `event_id` 和 `type`。如果每次重复都生成新 `event_id`，电脑端会认为是 3 个不同事件。
